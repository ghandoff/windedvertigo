# winded.vertigo — Collaboration Playbook

*How Garrett and Claude work together effectively. Learnings from 48+ sessions building this monorepo.*

## Session Patterns

### Starting a Session

1. Claude reads `docs/CLAUDE.md` for conventions and `memory/projects/creaseworks.md` for app state
2. Check `git status` and `git log --oneline -5` to orient
3. If continuing a compacted session, read the continuation summary carefully — it contains all prior context
4. Run `npx tsc --noEmit --project apps/creaseworks/tsconfig.json` if touching creaseworks

### Ending a Session

1. Update `memory/projects/creaseworks.md` with new migration numbers, file counts, feature status, latest commit hash
2. Commit and push any remaining changes
3. Note any pending tasks for the next session

### Session Types We've Identified

| Type | Pattern | Example |
|------|---------|---------|
| **Feature build** | Plan → implement → test → commit | User tiers, notification center, PWA |
| **Infrastructure** | Investigate → plan → implement → verify | Turborepo setup, Vercel migration |
| **Design/styling** | Review screenshot → adjust CSS → verify | Bio photo resize, a11y contrast fixes |
| **CMS wiring** | Create Notion DB → build sync script → wire to app | `/what/` page, footer sync |
| **Bug fix** | Reproduce → diagnose → fix → test | Footer HTML injection, Notion sync errors |
| **Exploration** | Investigate architecture → document findings → recommend | Deployment cost analysis, Supabase eval |

## What Works Well

### Planning Before Building
- For non-trivial features, we use plan mode to explore the codebase, design the approach, and get Garrett's approval before writing code
- Plans include file manifests, dependency analysis, and expected impact
- This prevents wasted effort and catches wrong assumptions early

### Commit Discipline
- Every meaningful change gets its own commit with a descriptive message
- Commit messages follow `type: summary` format (feat, fix, docs, chore)
- We commit frequently rather than batching — easier to revert if something breaks

### Verification Steps
- Run TypeScript compiler check after creaseworks changes
- Run tests when available (`npm run test`)
- Smoke test deployments when possible
- Check Vercel dashboard for deploy status

### CLI Over Dashboard
- Prefer terminal commands and scripts over clicking through browser UIs
- Faster, more reliable, and reproducible
- Exception: Vercel dashboard for checking deployment status/logs

## What to Watch For

### Don't Assume — Verify
- Always check the actual codebase before assuming file locations or content
- The monorepo has 6 apps — make sure you're editing the right one
- Package names in `package.json` don't always match directory names (e.g., `creaseworks` vs `@windedvertigo/harbor`)

### Dependency Awareness
- Changes to `packages/tokens` affect creaseworks AND harbor (Turborepo graph)
- Changes to root config (`turbo.json`, root `package.json`) trigger ALL projects
- Run `npx turbo run build --dry` to verify what would rebuild

### Notion Sync Auto-Commits
- The daily sync-notion workflow (6 AM UTC) commits directly to `main`. If you're mid-session and `git push` is rejected, run `git pull --rebase origin main` before retrying.
- The sync workflow also triggers on push to `main` when `scripts/` files change — be aware of cascading pushes.

### Deployment Limits
- Vercel free tier: 100 deploys/day shared across all 6 projects
- Turborepo + turbo-ignore reduces this, but root config changes still trigger all 6
- If hitting the limit, wait for daily reset rather than force-deploying

### Static Site Quirks
- `apps/site/` has no build step — it's raw HTML/CSS/JS served by Vercel
- Data files in `apps/site/data/` are auto-generated by sync scripts — don't edit directly
- Footer HTML is injected into all 10 pages from `packages/tokens/footer.html`

## Communication Preferences

### Garrett Prefers
- **Concise over verbose**: Get to the point, show the result
- **Show, don't tell**: Screenshots, code output, diff summaries
- **Ask before destroying**: Always confirm before deleting, overwriting, or irreversible actions
- **Explain the "why"**: Brief educational insights about implementation choices
- **Proactive problem-spotting**: Flag issues discovered during work (like the deployment quota problem)

### When to Ask vs. Proceed
- **Just do it**: Fixing typos, running status checks, reading files, exploring code
- **Ask first**: Deleting things, changing architecture, pushing to production, accepting external agreements
- **Use plan mode**: Multi-file features, infrastructure changes, anything with multiple valid approaches

## Naming Conventions

- **Branches**: `feat/feature-name`, `fix/bug-name`, `chore/task-name`
- **Commits**: `type: concise summary` — feat, fix, docs, chore, style, refactor
- **Files**: kebab-case for everything (`brand-guidelines.md`, `game-showcase.tsx`)
- **CSS variables**: `--wv-*` for brand, `--color-*` for semantic, `--space-*` for spacing
- **Package names**: `@windedvertigo/package-name` (scoped to org)
