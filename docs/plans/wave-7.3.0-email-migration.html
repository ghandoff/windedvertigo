Wave 7.3.0 · Planning Artifact

Email-as-Key Migration

Status Planning — no code yet  ·  Date 2026-04-22  ·  Gates Wave 7.2.0 · 7.3.1 · 7.4  ·  Burn-in 1 week in prod

TL;DR — Why this wave is the gate

Everything downstream in Wave 7 — shared workspace shell, /login, magic links, 2FA, the role-aware sidebar — assumes a stable, unique, deliverable identifier on every user record. Today that identifier is alias, which is mutable display-name. This wave promotes email to primary key across the Reviewers DB, adds a one-time confirmation banner for stragglers, teaches /api/auth/login to accept email, and seeds an email claim into the JWT. Alias stays everywhere as display-name; nothing gets deleted. One week of burn-in before Wave 7.2.0 is allowed to start.

§1. Context — why this wave, why now

Wave 7 reshapes the workspace around one door, many roles. The shape of that door — magic links for external reviewers, email + password for internal users, 2FA for super-user — assumes a single, stable, deliverable identifier on every user record. That identifier has to be email. You cannot send a magic link to an alias.

Today the Reviewers DB (b74c6186-d782-4499-85ac-3dee528a1977, data source 020a7d2e-1123-43a3-b90d-68f518c9787a) treats Alias as the uniqueness contract. Login resolves alias → reviewer and issues a JWT whose primary identity claim is also alias. Every sensitive code path — authenticatePcsWrite, admin checks, parseReviewerPage, /api/auth/login, /api/auth/register, the JWT claim set — is alias-keyed.

Master plan §3.3 names this the single most-likely-to-slip step in Wave 7 — not because it is technically hard but because it touches every user record and cannot be rolled forward if data is mis-backfilled. A wrong email binding locks a user out of their own account. That failure mode is why this wave gets a dry-run step, a confirmation banner, and a one-week burn-in before 7.2.0 is allowed to start.

Concrete consequences of not doing this wave first:

  • No magic links. Wave 7.3.1's /invite/{token} flow depends on a unique, present, deliverable email per reviewer.
  • No role-aware sidebar. Wave 7.4 needs an email-scoped localStorage namespace for sticky preferences. Alias is mutable; email is not.
  • No 2FA enrollment. TOTP / WebAuthn factors bind to the primary identifier.
  • No SSO ever. Every enterprise IdP (Okta, Azure AD, Google Workspace) expects email as the unique subject claim.

§2. Problem statement

Current auth trusts alias as the uniqueness key on the Reviewers DB. That assumption fails in three named ways:

  1. Alias collisions. As reviewers grow (Wave 8 community of practice is 30–50 external reviewers), first-name collisions are inevitable. sharon today, sharon2 tomorrow, sharon-k next year. No stable secondary key exists.
  2. Magic-link infrastructure needs a deliverable identity. Magic links are email → token → session. You cannot mail an alias.
  3. SSO is foreclosed. Every IdP binds on email. Migrating today is a no-regret move; migrating in three years with 50 reviewers and thousands of linked scores is a wire-and-splice job.

Silent fourth failure: audit log legibility. Alias-keyed audit entries rot as users rename themselves. Email is stable; alias is display-name.

§4. Migration phases

7.3.0a Read-only audit

Ship scripts/audit-reviewer-emails.mjs. No writes. Emits a diff report covering rows missing email, rows with duplicates, rows with malformed email (no @, no domain, trailing whitespace, comma, mixed-case domain), rows where alias looks reversed with email, rows where @nordicnaturals.com domain doesn't match alias local-part.

Output: stdout + tmp/audit-reviewer-emails-<ISO>.json. Exit 0 if clean, 1 if flags exist.

7.3.0b Resolve ambiguities

Human phase. Garrett + any affected user (probably 1–5 rows given 6 reviewers today) reconcile each flag. Missing emails get supplied. Duplicates get a canonical winner + deprecated sibling. Malformed emails get corrected via Notion or the backfill script.

Goal: audit script exits 0 on re-run.

7.3.0c Email Confirmation Banner

One-time blocking banner that appears on next login for any user whose Email is empty, or whose Email confirmed at is null, or whose email looks malformed. Posts to a new /api/auth/confirm-email endpoint which re-validates, pre-checks uniqueness, updates the reviewer, and stamps the confirmation timestamp.

Component: src/components/auth/EmailConfirmationBanner.js.

Draft copy (Garrett to finalize):
"We need to confirm your email before you keep going. Nordic Research is moving to email-based sign-in. This is the email we'll use for your magic-link invitations, password resets, and future sign-in."

7.3.0d Uniqueness constraint in code

Notion doesn't enforce DB-level uniqueness, so we enforce it in code. Every write path that touches Email runs a pre-write query. Duplicates throw EmailAlreadyInUseError; /api/auth/register maps that to a 409 with a legible message.

Race window is real (two simultaneous registrations could both pass pre-check) but acceptable at Nordic's write volume. The audit cron is the backstop.

7.3.0e Email login path

/api/auth/login accepts { identifier, password }. If identifier contains @, try email first, alias second. Else try alias first, email second. Everything else — password check, bcrypt migration, reset-grant, dual-token issue — is unchanged.

Login form field renames to "Email or alias" with a small subtitle: "Use your email going forward — alias login will retire in a future update."

7.3.0f JWT claim expansion

signAccessToken() adds email to the claim set alongside alias. Additive only; no consumer reads email from the JWT in 7.3.0. The claim is seeded for Wave 7.2.0 + 7.3.1 to consume without another auth round-trip.

7.3.0g One-week burn-in

No new code. One week in production, observing:

  • Login-failure rate (flat or down).
  • Banner-dismissal convergence toward 100% of MAU.
  • Zero EmailAlreadyInUseError events after first 48h.
  • Zero user-side reports of banner confusion.

§9. Rollback plan

Alias uniqueness is preserved throughout 7.3.0. Nothing we do removes alias as a working login key. Every phase is individually flaggable off:

  • WAVE_7_3_0_BANNER=false → banner renders null.
  • Revert pre-write check → creates still work; duplicate protection relaxes.
  • Ignore identifier on login → legacy alias path resumes.
  • Remove email claim → additive, no consumer breaks.

Irreversible point: Wave 7.2.0 is the first consumer that stops emitting alias-first semantics. 7.3.0 itself is fully reversible up to and including 7.3.0g. The burn-in window exists precisely to discover any reason to roll back before 7.2.0 starts.

§13. Open questions for Garrett

Q1

Shared mailboxes — forbid or support?

Husband-and-wife reviewer pair on one inbox. Cleanest answer: forbid — one email = one reviewer. Supporting shared inboxes tangles magic-link UX (whose session does the link create?) and muddies audit logs. But the call is yours before 7.3.0d ships.

Q2

Email case sensitivity — normalize to lowercase?

RFC 5321 technically case-sensitive, but every modern provider treats the local-part as case-insensitive. Recommend: lowercase on write, lowercase on compare. Side-effect: we canonicalize a display field for the user. Some users dislike that. Confirm before 7.3.0d.

Q3

Confirm-via-mail loop — require it in 7.3.0, or defer?

Strictest: banner submission triggers a one-time email link; email is verified only when clicked. Catches typos before they lock out the user on the next password reset. Cost is a new endpoint + template + second-step UX. Lean: defer — rely on the first successful magic-link in 7.3.1 as implicit verification. Genuine judgment call.

Q4

Banner + forced password reset — bundle or keep separate?

If a user's email and password-reset state are both stale, cleaner flow might be "confirm your email and rotate your password while you're here". Lean: keep separate — one banner, one job. But if 7.3.0a surfaces rows where both are stale, worth a quick re-think.
