3. cloud claude code vs local claude code

this is the single biggest source of confusion for the team, and it's also one of the most fixable.

what "cloud" claude code is

garrett opens claude.ai/code, types a task, and anthropic spins up a fresh virtual machine in their cloud. that VM:

  1. authenticates as garrett against github (via the claude github app he connected)
  2. clones the repository from github — not from his mac
  3. creates a new branch named claude/<adjective-name>-<hash>
  4. runs claude code in that VM. claude reads files, edits them, commits, and pushes the new branch to github
  5. opens a PR (or leaves a button to open one)
  6. the VM keeps running until the task is done or garrett closes it. monitorable from the claude mobile app
critical implication the cloud VM only sees what's on github. if garrett has local commits that haven't been pushed, the cloud session doesn't know they exist. it will start from whatever was on the github branch when the session began.

what "local" claude code is

garrett (or maria) runs claude in a terminal inside the repo on their mac. claude code:

  1. reads actual on-disk files, including uncommitted changes
  2. runs commands directly on the mac
  3. creates branches, commits, and pushes — but only when told to or approved
  4. can use git worktrees for parallel sessions

the "push before you delegate" rule

the one rule that fixes 70% of the confusion before starting a task in the cloud, or before maria starts a task on her mac: 
git status      # are there uncommitted changes?
git pull --rebase origin main   # take other people's changes first
git push        # publish your local work to github
without it: cloud claude works on stale code, work gets silently overwritten, maria pulls a confusing mix.

4. prs, branches, merging, pushing — the mental model

the three places code lives

working tree
files on disk in your editor
→
local repo
your .git folder
→
github (remote)
github.com/ghandoff/<repo>
  • working tree → local repo: git add then git commit
  • local repo → github: git push
  • github → local repo: git fetch (info only) or git pull (info + merge)
the thing people miss a commit on your mac is not on github until you push. maria can't see it. the cloud can't see it. pushing is publishing.

what a branch actually is

a branch is a pointer to a commit. nothing more. the word "branch" suggests a sprawling tree, but really it's a sticky note that says "this commit is the latest version of this work." your main branch is just another sticky note — the one that says "this is what we ship."

what a pr actually is

a PR is a conversation on github.com about merging one branch into another. it's not part of git — it's a github feature layered on top. it shows the diff, comments, CI checks, and a merge button. the lifecycle:

1. branch
→
2. push
→
3. open PR
4. review
→
5. merge
→
6. delete branch

the four merge styles

stylewhat it doeswhen to use
squash & mergecombines all commits in the PR into one commit on maindefault for the team. one PR = one commit on main. matches your recent merges (aabee6c feat(values-auction): Phase B-min (#83)).
merge commitpreserves every commit + adds a "merge" commitwhen the intermediate history matters (long-lived feature branch).
rebase & mergereplays branch commits onto main, no merge commitlinear history but with individual commits preserved. most teams don't bother.
fast-forwardjust moves main's pointer (only if main hasn't moved)almost never on a PR.

the standard PR flow

# 1. start clean
git checkout main
git pull --rebase origin main

# 2. create your branch (one branch per logical change)
git checkout -b feat/short-descriptive-name

# 3. work. commit often, with clear messages.
git add .
git commit -m "feat(scope): what you did"

# 4. push to github (the first time, set upstream)
git push -u origin feat/short-descriptive-name

# 5. open a PR. easiest way:
gh pr create --fill --base main

# 6. address review. push more commits — the PR updates.
git push

# 7. after merge, clean up.
git checkout main && git pull --rebase
git branch -d feat/short-descriptive-name
git push origin --delete feat/short-descriptive-name

6. worked scenarios

scenario a — maria edits a file, garrett edits it in cloud claude, both PRs open

setup. maria opens fix/harbour-tile-copy, changes apps/harbour/src/data/tiles.ts, pushes, opens a PR. garrett doesn't notice. he starts a cloud claude session "polish the harbour tile copy" — it branches from main (which doesn't yet contain maria's PR), edits the same file, opens its own PR.

result. two PRs touching the same file. whichever merges first wins; the second will conflict.

fix (process):

  1. check open PRs before starting a new task: gh pr list --state open.
  2. if you find maria's, review and merge first, or scope your cloud task differently.
  3. if both PRs exist: merge the smaller one first. on the other, git pull --rebase origin main, fix conflicts, push --force-with-lease.

fix (tooling): enable github's "require branches to be up to date before merging" branch protection on main.

scenario b — cloud claude PR looks weird

setup. garrett asked cloud claude to refactor apps/harbour/values-auction/src/state.ts. he had local changes to that file that weren't pushed. the cloud session worked on the old version.

fix.

  1. don't merge yet.
  2. push your local work to a different branch: git checkout -b fix/state-from-laptop && git push -u origin fix/state-from-laptop.
  3. on the cloud branch: git fetch && git checkout claude/<name> && git merge fix/state-from-laptop. resolve conflicts. push.
  4. now the PR includes both sets of changes. review and merge.

8. cross-device sync — desktop ↔ laptop ↔ phone

this section addresses garrett's specific question: "how do i sync my macbook pro to everything i've been working on at my mac mini, such that conversations persist across devices or i can reference them on my laptop?"

two separate things that get conflated

code & filesclaude conversations
where it livesgit repos in ~/Projects/~/.claude/ on each mac
how it syncsvia github (push / pull)doesn't, by default
visibility on phonegithub mobile apponly cloud sessions show in claude mobile app

the three sync strategies (use all three)

strategy 1 — git is the source of truth for code

before leaving the desk, push everything. on the laptop: git pull --rebase per repo. this is what the pre-mobile-handoff skill (section 9) automates.

# on the mac mini, before travel
cd ~/Projects/windedvertigo && git status && git push
cd ~/Projects/harbour-apps && git status && git push

# on the macbook pro at the destination
cd ~/Projects/windedvertigo && git fetch --all && git checkout <your-branch> && git pull --rebase

strategy 2 — .brain/handoff.md + .brain/TASKS.md are the source of truth for context

garrett already maintains .brain/TASKS.md and .brain/handoff.md in windedvertigo. these are git-tracked, so they sync via push/pull. that's where "what was i working on" lives, not in claude's chat history.

before travel, append a handoff note. on the laptop, the first thing claude reads is CLAUDE.md → which links to .brain/handoff.md → which tells claude what state to pick up in.

strategy 3 — cloud claude code sessions persist on anthropic's servers

sessions started at claude.ai/code keep running on anthropic's infrastructure even when garrett's machines are off. they're visible from:

  • any browser at claude.ai/code (logged in as garrett)
  • the claude mobile app (iOS / android)
  • the macbook pro at the destination

so for tasks garrett wants to "keep going while traveling", the right move is: kick them off as cloud sessions before leaving, not local sessions. then resume them from the phone in the airport, the laptop on the plane, or the destination browser.

strategy 4 (newer) — claude code remote control

as of february 2026, anthropic shipped remote control, a research-preview feature that bridges a local claude code terminal session with claude.ai/code, the iOS app, and the android app. the session keeps running on garrett's mac mini; the web/mobile UI is just a window into it. that means his CLAUDE.md, custom skills, file access, and MCP integrations all stay available — but he can interact with the same conversation from any device.

this is the closest thing to true "same conversation everywhere" today. enable it from the local claude CLI; it produces a QR code to connect a phone/browser.

recommended setup for garrett's two-mac + phone life

  1. before any travel: invoke the pre-mobile-handoff skill (section 9). it pushes everything and writes .brain/handoff.md.
  2. tasks you want to keep going: kick them off as cloud claude sessions at claude.ai/code (not local). monitor from the phone.
  3. on the macbook pro at the destination:
    • git fetch --all && git pull --rebase in each repo
    • let claude read CLAUDE.md + .brain/handoff.md for context
    • start a local claude session — it now has the same context as the desk
  4. if you want literal same-conversation continuity: use remote control on the mac mini so the laptop/phone are windows into the same session.
about chat history specifically the local ~/.claude/ conversation history is per-machine and does not sync by default. third-party tools (claude-sync, claude-session-sync via iCloud Drive) exist, but they're community-built and not officially supported. for now, treat conversation history as ephemeral and your .brain/ markdown files as the durable record.

does CLAUDE.md do the same thing?

no — they complement each other:

  • CLAUDE.md is always-on context. claude reads it at the start of every session. good for "here's what this repo is, here's our voice, here's our infra."
  • skills are invokable workflows. they only run when the user (or claude itself) decides the moment matches. good for "when X happens, do Y, Z, then W."

think of CLAUDE.md as the orientation packet, and skills as the recipes. you want both.

how to invoke a skill

you don't have to. just say the natural-language thing — claude reads the description and decides. for example:

# any of these will trigger pre-mobile-handoff:
"i'm about to head out — make sure everything is pushed"
"going mobile, can you sync?"
"i'm switching to my laptop, make sure maria's on the latest"

# if you want to be explicit:
"run the pre-mobile-handoff skill"

11. branch & worktree cleanup playbook

copy-paste these. read first.

step 1 — audit (read-only)

cd ~/Projects/harbour-apps
git fetch --prune
git branch -vv           # local branches, ': gone' = upstream deleted
git branch -r --merged origin/main  # remote branches already merged
gh pr list --state open --json headRefName,number,title --limit 100

step 2 — delete merged claude/* branches (remote)

# dry-run first:
git branch -r --merged origin/main | grep "origin/claude/" | sed 's|origin/||'

# then delete:
git branch -r --merged origin/main | grep "origin/claude/" | sed 's|origin/||' | \
  xargs -n1 -I{} git push origin --delete {}

step 3 — delete merged local branches

git checkout main && git pull --rebase
git branch --merged main | grep -v "^\* " | grep -v "^  main$" | xargs -n1 git branch -d
git branch -vv | awk '/: gone]/{print $1}' | xargs -r git branch -D

step 4 — prune orphaned worktrees

git worktree list
git worktree prune --dry-run -v
git worktree prune

one-command alias (zsh / bash)

git-cleanup() {
  git fetch --prune
  git checkout main && git pull --rebase
  echo "== local merged =="; git branch --merged main | grep -v "^\*\|^  main$"
  echo "== local 'gone' =="; git branch -vv | awk '/: gone]/{print $1}'
  echo "== prunable worktrees =="; git worktree list | grep prunable
  read -p "proceed? (y/N) " yn
  if [[ "$yn" == "y" ]]; then
    git branch --merged main | grep -v "^\* \|^  main$" | xargs -r git branch -d
    git branch -vv | awk '/: gone]/{print $1}' | xargs -r git branch -D
    git worktree prune
  fi
}

or just say "clean up branches" and let the branch-cleanup skill (section 9) handle it.

13. cheat-sheet

the five rules

  1. push before you delegate. cloud claude sees github, not your mac.
  2. one branch per PR per intent. don't pile fixes onto a feature branch.
  3. pull main daily.
  4. merge fast or delete fast. branches older than a week are decisions or noise.
  5. migrations are serialized. only one schema change in flight at a time.

daily commands

git status                      # where am i?
git fetch --prune               # refresh remote info
git pull --rebase               # take main's changes
git checkout -b feat/foo        # start work
git push -u origin feat/foo     # publish
gh pr create --fill             # open PR
gh pr status                    # your open PRs

when something's wrong

git log origin/main..HEAD             # committed but not pushed
git log HEAD..origin/main             # on remote but not local
git stash                             # put aside uncommitted work
git stash pop                         # bring it back
git reflog                            # time machine — find lost work
