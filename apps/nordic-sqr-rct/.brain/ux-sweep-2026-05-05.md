# UX papercut sweep — 2026-05-05

Pre-Monday audit of /pcs/{documents, claims, ingredients} authoring
surfaces, looking for papercuts of similar character to the four
already fixed in the Evidence Library today.

## Methodology

Walked all PCS authoring pages and their shared `PcsTable` component,
comparing each against the four heuristics that produced today's
Evidence Library fixes (default sort sanity, empty states, loading
skeleton parity, post-write refresh) plus the broader heuristic list
in the brief. For each page I read the list view, the detail view,
and (where present) the create flow, focusing on the header/title
band, the table or content body, the empty state, and the data-fetch
path. PcsTable itself was diffed against its callers to spot pages
that fail to opt into recently-added features (defensive unwrapping,
defaultSort props). Findings below are limited to issues an actual
researcher would hit during Monday onboarding.

## Findings — by severity

### Severity: ship-blocker (would actively confuse a researcher Monday)

1. **Default sort missing on Documents, Claims, Ingredients** — heuristic: *Default sort sanity*.
   `src/app/pcs/documents/page.js:313`, `src/app/pcs/claims/page.js:158`, `src/app/pcs/ingredients/page.js:157`.
   Only `evidence/page.js` passes `defaultSortKey="lastEditedTime"`/`defaultSortDir="desc"` to `PcsTable` (lines 186–187). The other three call sites omit these props, so `PcsTable.js:47–48` defaults to `sortKey=null` and rows arrive in API order. Same papercut the Evidence Library just fixed — when Sharon edits a document, the freshly-touched row sinks back into the alphabetic middle. Fix: add the same two props to each call site (one-line change per file).

2. **Claims list crashes on API error** — heuristic: *Stale-data UX / defensive coercion*.
   `src/app/pcs/claims/page.js:35`. `setClaims` is called with raw `res.json()` output. If the API returns `{ error: '…' }` (it does on capability-denied responses) `PcsTable.js:88` will throw `data.filter is not a function` and white-screen the page. Evidence already fixed this (`evidence/page.js:40` — "defensively unwrap"). Documents and Ingredients are fine because they wrap in `Array.isArray()`. Fix: same `Array.isArray(data) ? data : (data?.items || [])` coercion on line 35.

3. **Ingredients detail page has no edit affordances at all** — heuristic: *Action affordance visibility / capability gating*.
   `src/app/pcs/ingredients/[id]/page.js:23–24`. `useAuth()` is imported then immediately `void user;` with the comment "reserved for future inline edits". Every field — canonical name, category, FDA RDI, regulatory ceiling, bioavailability notes, interaction cautions, synonyms — is read-only with no Edit button anywhere on the page. PCS_WRITERS will reasonably expect to be able to fix a typo Monday and find no surface for it. Fix (deferred is acceptable, but at minimum surface the limitation): if inline edit isn't shipping Monday, render a one-line "This page is read-only — edit ingredients via the import flow" so writers don't hunt for a missing pencil.

### Severity: high (worth fixing this week)

4. **Loading skeletons don't match real layout on three of four pages** — heuristic: *Loading state*.
   `src/app/pcs/documents/loading.js`, `src/app/pcs/claims/loading.js`, `src/app/pcs/evidence/loading.js` — all render `h-8` title + 8–10 stacked `h-10` row bars. But the real pages render a stat strip (Documents has 6 boxes), a banner card (Documents view banner, Claims review-queue banner), or a search panel (Evidence). The skeleton-to-real-layout jump is jarring and makes the page feel like it's rebuilding itself. Ingredients (`ingredients/loading.js:5`) renders a grid of cards which doesn't match its actual list view either. Fix: align each skeleton to the page's first-paint shape (title + stat strip + table-row stack for Documents; title + banner + table for Claims; title + search panel + table for Evidence; title + filter row + table for Ingredients).

5. **Documents and Ingredients have no "New X" affordance on the list page** — heuristic: *Cross-page navigation / action affordance visibility*.
   `documents/page.js:264–280` and `ingredients/page.js:127–155` have header rows with no create button. Claims has `/pcs/claims/new` but the list page (`claims/page.js:130–139`) doesn't link to it either — operators have to know the URL. Documents creation lives at `/pcs/data?tab=imports` (per `claims/new/page.js:73`), Ingredients creation has no surfaced path at all. Researchers will scan for a "+ New" button and find nothing. Fix: add a primary-styled CTA in each list header — Documents → `/pcs/data?tab=imports`, Ingredients → wherever the import flow is, Claims → `/pcs/claims/new`.

6. **Claims list URL filters aren't surfaced as UI** — heuristic: *Filter discoverability*.
   `src/app/pcs/claims/page.js:20–21,111–115`. The page reads `?bucket=`, `?noEvidence=`, `?versionId=` and changes the title accordingly, but there's no filter UI to *enter* those modes from the page. The only way to land in "Bucket 3A" is by clicking through from another page that already constructs the URL. Compare to Documents (`documents/page.js:272–279`) which surfaces all six views as a clickable Stat strip. Fix: add a small filter row above the table (3 bucket buttons + an "Evidence gaps" toggle).

7. **Ingredients page renders two filter inputs stacked** — heuristic: *Filter discoverability / visual noise*.
   `src/app/pcs/ingredients/page.js:137–143` renders a "Search name or synonyms..." input, then `PcsTable.js:132–138` renders its own "Filter..." input directly below it. Two boxes that both filter the table, no labels, no indication they're different scopes (page-level vs. row-level string match). Fix: either (a) pass a flag to PcsTable to suppress its built-in filter when the parent provides one, or (b) hide the page-level search since PcsTable's filter already searches across all visible columns.

8. **Claims detail header can collide on long claim text** — heuristic: *Title/header collisions* (the same heuristic that hit Evidence detail today).
   `src/app/pcs/claims/[id]/page.js:655–681`. The wrapper is `flex items-start justify-between` but the title `<div>` (line 656) lacks `min-w-0`, and `h1` (line 664) lacks `break-words`. Long claim text + History button + bucket pill on the right will overlap on viewports under ~900px. Evidence detail fixed this earlier today with `min-w-0 flex-1` + `break-words` (`evidence/[id]/page.js:327,331`) — Claims detail still has the bug. Fix: add `min-w-0 flex-1` to the title container and `break-words` to the `h1`.

9. **PcsTable filter input has no label or aria-label** — heuristic: *Accessibility basics*.
   `src/components/pcs/PcsTable.js:132–138`. The filter `<input>` only has a placeholder ("Filter..."). Screen readers announce it as just "edit text". Multiple PcsTable instances per page (e.g., document detail with the Versions sub-table) compound this. Fix: add `aria-label="Filter rows"` and consider a visible `<label>` so the affordance is clear without focus.

10. **Documents stat strip uses color-only active signaling** — heuristic: *Accessibility basics / color-only signaling*.
    `src/app/pcs/documents/page.js:319–333`. The `Stat` button distinguishes "active view" by border-color + ring (`border-pacific-400 ring-1 ring-pacific-200`). No `aria-pressed` or `aria-current`, and the styling difference is subtle for low-vision users. Fix: add `aria-pressed={active}` to the `Cmp` (when it's a button), and consider darkening the active border or adding a left accent bar.

### Severity: low (file as future polish)

11. **Inline "Edit" text-button hit area is sub-spec on claim detail** — heuristic: *Click-target sizing*.
    `src/app/pcs/claims/[id]/page.js:70–76,786,914`, plus the row-level `Edit` in `EvidencePacketRow` (line 511). These are bare `<button class="text-xs text-pacific-600">Edit</button>` with no padding — measured tap target is roughly 14px tall on Retina. Recommend `px-1.5 py-0.5` minimum; the `History` button next to them already has appropriate sizing.

12. **"RA Review Actions" gated only on `canWrite` (PCS_WRITERS), not on RA-specific role** — heuristic: *Capability gating*.
    `src/app/pcs/claims/[id]/page.js:751–779`. The header reads "RA Review Actions" with Approve/Reject/Request More Evidence buttons, but visibility uses `hasAnyRole(user, ROLE_SETS.PCS_WRITERS)` (line 544), which includes Research-team roles per the project memory note (Research = Sharon/Gina/Adin/Lauren; RA = the 2 TBD). Sharon will see "Approve Claim" buttons that probably aren't hers to click. Fix: gate on an RA-specific role set if one exists, or rename the section header to be writer-neutral ("Status quick-set") so the role/UI mismatch is gone.

13. **Documents list empty state is a dead end** — heuristic: *Empty state*.
    `src/app/pcs/documents/page.js:308–311`. When a view is empty (e.g. "Has Duplicates" with 0 rows), the body just says "No documents in this view." with no next action. Compare to Claims detail's empty-evidence state (`claims/[id]/page.js:924–935`) which calls out the gap and offers a primary CTA. Fix: add a "← Back to Active" or "+ Import documents" CTA inside the empty card.

14. **Ingredients detail Forms table has no canEdit / no add-form button** — heuristic: *Action affordance visibility*.
    `src/app/pcs/ingredients/[id]/page.js:153–193`. The Forms section header (line 154–161) has `flex items-center justify-between` with an empty right-hand slot — looks like it was intended to host an "+ Add form" button that never landed. Fix: either drop the `justify-between` (cosmetic) or wire the add-form action.

15. **`Field` component in evidence detail doesn't render `options` prop in non-edit mode** — heuristic: *Stale-data UX (cosmetic)*.
    `src/app/pcs/evidence/[id]/page.js:13–92`. Minor: in non-edit mode the displayed value for a `select` field is the raw stored string with no validation hint. If the stored value drifts from `EVIDENCE_TYPES`, the user sees garbage with no signal. Low priority.

## Pages audited but found clean

- `src/app/pcs/documents/[id]/view/page.js` — Living view skeleton mirrors `loading.js` on client refetches (`page.js:48–58`). Good pattern.
- `src/app/pcs/claims/new/page.js` — Form/upload tabbed entry; the form path correctly routes to `/pcs/claims/${id}` after submit (line 148). One nit: defaults to "Upload" tab (line 25) even though the Form tab is marked "live" (line 63) — minor.
- `src/components/pcs/PcsTable.js` — Sort persistence keyed per user+table, defensive against SSR mismatch, properly versioned (`pcs-sort-v2-` after the defaultSort change). Solid.
- `src/app/pcs/documents/[id]/page.js` header band — uses `flex items-center justify-between` with a short `pcsId` title; the four-button right cluster fits because the title is bounded. Less robust than Evidence/Claims detail patterns but works in practice.

## Patterns worth replicating

- **Evidence Library's two-phase optimistic update + delayed re-fetch** (`evidence/page.js:123–139`) — the comment ("Notion's Database query API can take 2–5s") generalizes to every list that mutates via Notion. Documents and Ingredients list pages don't have a "create from this page" path yet, but when they get one, this pattern should be the template.
- **Documents view banner with "← Back to Active"** (`documents/page.js:282–295`) — a clean breadcrumb-out for filtered views. Claims has the same shape ("Show all" link, line 134–138) but inline; the banner version reads better when the filter changes the page semantics.
- **Claim detail's empty-evidence call-out** (`claims/[id]/page.js:924–935`) — red-tinted empty state that names the problem ("evidence gap") and embeds the CTA. This is the gold standard for the rest of the empty states on these pages.
- **InlineField pencil-always-visible pattern** (`documents/[id]/page.js:90–129`, comment dated 2026-05-04) — explicitly avoids opacity-0 group-hover so the editability is discoverable. The Claims detail "Edit" text-buttons (finding #11) should adopt this affordance.
